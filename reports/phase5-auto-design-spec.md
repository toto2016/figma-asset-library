# Phase 5: 需求描述 → 自动出图 — 技术方案定义

> 基于 12 人工程评审中 AI/ML 工程师和产品经理的建议

---

## 一、三级出图能力定义

| 级别 | 名称 | 输入 | 输出 | 精度 | 技术路径 |
|------|------|------|------|------|---------|
| L1 | 线框图 | 自然语言需求 | 灰色线框布局 | 结构正确 | LLM → 组件选择 → 布局代码 |
| L2 | 组件拼装 | 需求 + 页面类型 | 用真实组件拼装的页面 | 视觉 80% 还原 | RAG 检索 → 组件实例化 → Auto Layout |
| L3 | 像素级还原 | 需求 + 品牌规范 | 可直接交付的设计稿 | 视觉 95%+ | L2 + Token 绑定 + 内容填充 |

---

## 二、L1 线框图方案

### 流程

```
用户需求（自然语言）
  ↓
LLM 理解需求 → 识别页面类型（支付/KYC/首页...）
  ↓
从 classified-v2 索引检索匹配的组件列表
  ↓
生成布局描述（JSON 结构）
  ↓
use_figma → 在 Figma 中创建灰色矩形布局
```

### 所需数据
- `catalog/classified-v2/` 多维度索引（已有）
- 组件尺寸数据（部分已有，在 deep extraction 中）
- 页面模板定义（需新增）

### 预估 MCP 调用
- 每个页面: 1 次 `use_figma`
- 预估精度: 结构 70%+

---

## 三、L2 组件拼装方案

### 流程

```
用户需求 + 页面类型
  ↓
RAG 检索: 需求 → Embedding → 匹配组件语义描述
  ↓
组件选择 + 排列规划（参考 Handover 页面的组件共现数据）
  ↓
use_figma → 实例化真实组件 + Auto Layout 排列
  ↓
绑定 Semantic Token（颜色/字体）
```

### 所需数据
- 组件语义描述（需新增 — 本次生成）
- 组件共现关系图（从 Handover 页面提取）
- 组件缩略图嵌入向量（需生成）
- 页面类型模板（支付流/KYC流/首页...）

### 预估 MCP 调用
- 每个页面: 2-3 次 `use_figma`（创建 Frame + 实例化组件 + 调整布局）
- 预估精度: 视觉 80%

---

## 四、L3 像素级还原方案

### 额外要求（基于 L2）

1. **Token 绑定**: 所有颜色/字体使用 Variables，支持 Mode 切换
2. **内容填充**: 真实的文案/图片/数据（需接入内容源）
3. **响应式**: Compact/Default spacing 适配
4. **品牌一致性**: 使用 Atome 品牌色和字体

### 所需数据
- Variables Collection（迁移方案已有，需实施）
- 内容模板（按业务场景预设文案/图片）
- 断点定义和响应规则

### 预估 MCP 调用
- 每个页面: 5-8 次 `use_figma`
- 预估精度: 95%+

## 五、技术路径选择建议

| 维度 | L1 线框图 | L2 组件拼装 | L3 像素级 |
|------|----------|-----------|----------|
| 实现难度 | ★★☆ | ★★★☆ | ★★★★★ |
| 前置条件 | 已就绪 | 需语义描述+共现图 | 需 Variables+内容源 |
| 业务价值 | 中（加速沟通） | 高（减少 50% 设计时间） | 极高（近乎替代初稿） |
| MCP 消耗 | 低 | 中 | 高 |
| 建议顺序 | **第一步** | **第二步** | **第三步** |

### 推荐路线

1. **Phase 5a（1 周）**: 实现 L1 线框图
   - 定义 5 个页面模板（首页/支付/KYC/商户/设置）
   - 编写模板 → Figma 的转换脚本
   - 在 Atome AI Test 中验证

2. **Phase 5b（2 周）**: 实现 L2 组件拼装
   - 生成组件语义描述（本次完成）
   - 建立组件共现关系
   - 开发 RAG 检索 + 组件实例化管线

3. **Phase 5c（长期）**: 探索 L3 像素级
   - 依赖 Phase 4 统一组件库和 Variables 完成
   - 需要更多 MCP 调用额度
