## 一、工具全景测试结果

共 17 个工具，全部测试完毕。按功能分为 5 类：

### 1. 读取类工具（6 个）

| 工具 | 计费 | 状态 | 测试结果 |
|------|------|------|---------|
| `get_metadata` | 是 | 成功 | 返回 XML 格式节点结构（ID/类型/名称/位置/尺寸） |
| `get_design_context` | 是 | 成功 | 返回参考代码 + 截图 + 上下文元数据，触发 Code Connect 提示 |
| `get_screenshot` | 是 | 成功 | 返回高质量节点截图，可用于资产目录 |
| `get_variable_defs` | 是 | **失败** | 报错"需要先选择图层"，似需桌面端交互 |
| `search_design_system` | 是 | 成功 | 搜索"button"返回 20 组件 + 18 变量 + 2 样式，跨多个库 |
| `get_figjam` | 是 | 成功 | 返回 FigJam 画布 XML + 截图 |

**核心发现**:
- `get_metadata` 是轻量扫描利器，只返回结构，不含设计细节
- `get_design_context` 是最完整的读取工具，但会主动引导 Code Connect 映射
- `get_variable_defs` 在 MCP 环境下存在限制，无法直接使用
- `search_design_system` 能搜索到组织内所有已发布库的资产，信息非常丰富

### 2. 写入类工具（3 个）

| 工具 | 计费 | 状态 | 测试结果 |
|------|------|------|---------|
| `use_figma` | 是 | 成功 | 成功创建变量集合、Button 组件、Card 布局 |
| `create_new_file` | 是 | 成功 | 成功创建 Design 和 FigJam 两种文件类型 |
| `create_design_system_rules` | 是 | 成功 | 返回代码库分析提示模板（不直接操作 Figma） |

**核心发现**:
- `use_figma` 是最强大的工具，可执行任意 Plugin API 代码
- **DROP_SHADOW 需要 blendMode 字段**（文档未明确说明，已踩坑记录）
- 创建文件必须指定 `planKey`，且**必须用组织 key** 才能享受 Full seat 额度
- `create_design_system_rules` 实际是一个提示生成器，不消耗 Figma 资源

### 3. Code Connect 工具（5 个）

| 工具 | 计费 | 状态 | 测试结果 |
|------|------|------|---------|
| `get_code_connect_suggestions` | 是 | 成功 | 检测到未映射的 Button 组件 |
| `add_code_connect_map` | **免费** | 失败 | 组件未发布到库，无法映射 |
| `get_context_for_code_connect` | 是 | 成功 | 返回组件属性定义和后代树结构 |
| `get_code_connect_map` | 是 | 成功 | 返回空对象（无现有映射） |
| `send_code_connect_mappings` | 是 | 未测 | 需先有 suggestions 才能批量保存 |

**核心发现**:
- Code Connect 需要组件**先发布到库**，才能建立代码映射
- 映射后 `get_design_context` 会返回对应代码组件引用
- 这套工具链适合 Phase 5（统一组件库建设后的代码映射）

### 4. FigJam 工具（2 个）

| 工具 | 计费 | 状态 | 测试结果 |
|------|------|------|---------|
| `generate_diagram` | 是 | 成功 | 通过 Mermaid 语法生成流程图，自动创建 FigJam 文件 |
| `get_figjam` | 是 | 成功 | 读取 FigJam 画布内容 |

**核心发现**:
- `generate_diagram` 适合生成文档和流程图，但与资产提取无直接关系
- 支持 flowchart / sequence / state / gantt 四种图表类型

### 5. 免费工具（3 个）

| 工具 | 计费 | 状态 | 测试结果 |
|------|------|------|---------|
| `whoami` | 免费 | 成功 | 确认身份、计划、席位类型 |
| `generate_figma_design` | 免费 | 成功 | 返回网页捕获选项（newFile/existingFile/clipboard） |
| `add_code_connect_map` | 免费 | 失败 | 需已发布组件 |

## 二、关键能力矩阵

针对项目五个阶段的工具适配度：

| 能力需求 | 主要工具 | 适配度 | 说明 |
|---------|---------|--------|------|
| 批量扫描文件结构 | `get_metadata` | ★★★★★ | 轻量 XML，适合 914 文件全量扫描 |
| 提取组件信息 | `use_figma` | ★★★★★ | 可遍历所有页面和节点 |
| 截取视觉资产 | `get_screenshot` | ★★★★☆ | 需逐节点截取，大批量时调用数多 |
| 读取设计变量/Token | `use_figma` | ★★★★★ | `get_variable_defs` 不可用，但 Plugin API 可直接读取 |
| 搜索已有设计系统 | `search_design_system` | ★★★★★ | 跨库搜索能力强大 |
| 创建统一组件库 | `use_figma` + `create_new_file` | ★★★★☆ | 可编程创建，但复杂组件需多次调用 |
| 代码-设计映射 | Code Connect 工具链 | ★★★☆☆ | 需先发布组件，流程较长 |
| 网页到 Figma | `generate_figma_design` | ★★★★☆ | 免费且强大，适合反向导入 |
| 自动出图 | `use_figma` + `search_design_system` | ★★★★☆ | 搜索复用 + 编程生成组合 |

## 三、踩坑记录

| # | 问题 | 原因 | 解决方案 |
|---|------|------|---------|
| 1 | 个人团队 Starter 限额耗尽 | POC 文件在个人团队下 | 用组织 key 创建新文件 |
| 2 | `DROP_SHADOW` 设置报错 | 缺少 `blendMode` 字段 | 添加 `blendMode: 'NORMAL'` |
| 3 | `get_variable_defs` 报错 | "需要先选择图层" | 改用 `use_figma` + Plugin API 读取变量 |
| 4 | `add_code_connect_map` 失败 | 组件未发布到库 | 需先将组件 publish 为库 |
| 5 | 失败的调用也消耗额度 | MCP 计费机制 | 脚本必须先验证再执行 |
| 6 | `get_design_context` 触发 CC 提示 | 检测到未映射组件 | 需用 `disableCodeConnect: true` 跳过 |

## 五、多角色可行性评估

### 角色 1：UED 设计主管

**关注点**: 设计资产完整性、设计系统一致性、团队效率提升

**评估**:
- **可行性: 8/10**
- 优势: MCP 能系统化提取 12 个项目的全量资产，解决设计资产碎片化问题。
  `search_design_system` 已证明可跨库搜索组件/变量/样式，这对审计现有设计系统非常有价值。
- 风险: 914 个文件中可能存在大量废弃/过时设计，自动提取会引入噪声。
  需要人工标注资产状态（活跃/废弃/归档）。
- 建议: 先从 P0 项目（Flow Library + DesignOps, 126 文件）验证提取质量，
  再决定是否全量展开。同时安排设计师参与资产分类标注。

### 角色 2：前端架构师

**关注点**: 代码-设计一致性、组件复用率、开发效率

**评估**:
- **可行性: 7/10**
- 优势: Code Connect 工具链可建立 Figma 组件到代码组件的双向映射，
  `get_context_for_code_connect` 能自动解析组件属性和变体结构。
  未来 `get_design_context` 可直接返回对应的代码组件引用。
- 风险: Code Connect 要求组件**必须先发布到库**，这意味着需要先完成
  统一组件库建设（Phase 4），才能实现代码映射（Phase 5）。
  且当前 Atome 前端组件库与 Figma 组件的对应关系需要人工梳理。
- 建议: 并行推进——资产提取和组件库建设用 MCP 自动化，
  代码映射关系由前端开发协助定义。

### 角色 3：项目经理（PMO）

**关注点**: 时间线、资源投入、ROI、风险管控

**评估**:
- **可行性: 6/10**
- 优势: MCP 调用完全可编程，可在非工作时间自动执行批量提取，
  无需占用设计师人力。200 次/天的额度可 pipeline 化使用。
- 风险:
  - 预估 3000-5000 次调用，需 **15-25 个工作日**完成全量提取
  - 失败调用也消耗额度（今天就浪费了 2 次）
  - Figma 可能调整 API 限额或定价
  - 文件权限为 View（只读），无法修改源项目
- 建议:
  1. 设立每日额度预算和监控告警（超 150 次/天时预警）
  2. 按 Wave 分批上线，每 Wave 结束做 checkpoint 验收
  3. 准备降级方案（手动提取关键资产）

### 角色 4：DevOps / 平台工程师

**关注点**: 自动化可行性、系统集成、可靠性

**评估**:
- **可行性: 7/10**
- 优势:
  - MCP 通过 Cursor/Claude 调用，脚本化程度高
  - `use_figma` 可执行任意 JS，灵活性极强
  - 可用 `return` 结构化返回数据，便于后续处理
- 风险:
  - MCP 调用必须通过 IDE（Cursor），无法纯 CI/CD 执行
  - 无 REST API 回退方案（依赖 MCP 协议）
  - 页面上下文每次调用重置，跨页面操作需额外管理
  - 速率限制（15 次/分钟）限制并发
- 建议:
  1. 构建本地缓存层，已提取的数据不重复请求
  2. 实现幂等提取脚本（重跑不会重复创建）
  3. 每次调用后立即持久化结果到本地 JSON
  4. 考虑 Figma REST API 作为补充（需单独申请 token）

### 角色 5：数据 / AI 工程师

**关注点**: 数据质量、索引结构、AI 可用性

**评估**:
- **可行性: 8/10**
- 优势:
  - `use_figma` 可深度遍历节点树，提取完整属性链
  - `get_screenshot` 提供视觉数据，可构建多模态索引
  - `search_design_system` 证明跨库搜索已有成熟实现
  - 变量/Token 可通过 Plugin API 完整提取（绕过 `get_variable_defs` 限制）
- 风险:
  - 914 文件的节点总量可能达百万级，需设计采样策略
  - 截图存储量大（每个关键节点一张）
  - 非结构化数据（设计师命名不规范）需清洗
- 建议:
  1. 先提取文件级元数据（页面数/组件数/变量数），构建"文件画像"
  2. 定义资产类型 schema（组件 / 页面模板 / 图标 / 插图）
  3. 截图只取关键节点（组件根节点 + 页面顶层 frame）
  4. 构建向量索引时，同时嵌入节点路径 + 视觉特征

### 角色 6：信息安全官 (CISO)

**关注点**: 数据安全、权限合规、第三方风险

**评估**:
- **可行性: 9/10（安全角度无重大阻断）**
- 优势:
  - View 权限只读，无法修改源项目文件
  - 写入操作限制在 Atome AI Test 项目
  - MCP 通过 Figma 官方服务器，非第三方代理
  - 提取的数据存储在本地，不经过外部存储
- 风险:
  - 本地 JSON 索引含设计细节，需控制访问权限
  - 截图资产可能含未发布的产品设计，需注意泄露
  - AI Agent 执行的代码在 Figma 服务器端运行
- 建议:
  1. `screenshots/` 目录已在 `.gitignore`，保持不提交
  2. 索引文件不含像素级设计数据，风险可控
  3. 定期审计 MCP 调用日志

## 六、总体结论

### 可行性评分：7.5 / 10

**核心优势**:
1. `use_figma` + `get_metadata` + `get_screenshot` 三件套覆盖 90% 提取需求
2. `search_design_system` 开箱即用，跨库搜索能力超预期
3. 200 次/天额度可支撑项目，虽需 15-25 天但可接受
4. 全流程可编程，可复用、可审计

**主要风险**:
1. 失败调用也消耗额度，脚本质量至关重要
2. `get_variable_defs` 无法使用，需 `use_figma` 替代方案
3. 依赖 Cursor IDE 运行，无法纯后端自动化
4. 914 文件全量提取的时间和资源投入需管理层支持

**下一步行动**:
1. 启动 Wave 1（P0 项目），验证批量提取流水线
2. 建立每日调用监控和预算管理
3. 设计资产分类 schema 和索引结构
4. 与前端团队对齐 Code Connect 映射需求
